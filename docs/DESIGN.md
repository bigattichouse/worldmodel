# WorldModel: Executable Reasoning LLM

**Version:** 3.0
**Status:** Active Design
**Goal:** Fine-tune a small LLM to reason, design, write executable code, observe results, and continue reasoning — all within a single generation pass.

---

## 1. Vision

Most LLMs generate plausible-sounding text. This project trains a model that **verifies its own reasoning through execution**. When asked "what is the trajectory of a ball thrown at 45°?", the model doesn't guess — it writes code, runs it, reads the output, and uses actual numbers in its answer.

The key insight: code execution is a grounding mechanism. It forces correctness on quantitative reasoning and provides feedback the model can use within the same generation.

---

## 2. Token System

Four special tokens structure the generation. Three are generated by the model; one is injected by the runtime.

```
<think>
Abstract reasoning about the problem. What is being asked?
What approach makes sense? What are the steps?
</think>

<model>
Design of the solution. This is lightweight and flexible:
- Could be a single phrase: "Use the quadratic formula"
- Could be pseudocode: "loop over items, accumulate sum"
- Could be full Blueprint notation for complex system designs
- Optional for simple problems
</model>

<code>
# Python code to execute
# Must be complete and runnable
result = some_calculation()
print(result)
</code>

<output>
[INJECTED BY RUNTIME — NEVER GENERATED BY MODEL]
Actual stdout/stderr from executing the code block above
</output>
```

### 2.1 Rules

- **`<think>`**: Internal reasoning. Not shown to user by default. Can appear multiple times.
- **`<model>`**: Optional design step. Use for problems that benefit from explicit planning. Omit for simple arithmetic.
- **`<code>`**: Python. Must `print()` whatever the model wants to see. Runtime captures stdout.
- **`<output>`**: Injected verbatim by the execution engine after running `<code>`. The model never generates this tag — training data shows it appearing as ground truth.
- **Multiple cycles**: All four tokens can repeat. Each `<code>/<output>` cycle feeds into the next `<think>`.

### 2.2 Example: Simple Arithmetic

```
User: What is the compound interest on $1000 at 5% for 3 years?

<think>
Compound interest formula: A = P(1 + r)^t
P=1000, r=0.05, t=3
</think>
<code>
P = 1000
r = 0.05
t = 3
A = P * (1 + r)**t
interest = A - P
print(f"Final amount: ${A:.2f}")
print(f"Interest earned: ${interest:.2f}")
</code>
<output>
Final amount: $1157.63
Interest earned: $157.63
</output>
After 3 years at 5% annual compound interest, $1000 grows to **$1157.63**, earning **$157.63** in interest.
```

### 2.3 Example: Multi-Step Reasoning

```
User: A projectile is launched at 30° with initial velocity 20 m/s. How far does it travel?

<think>
Need range formula, or I can simulate it step by step.
Use kinematics: separate x and y components.
</think>
<model>
- vx = v * cos(θ), vy = v * sin(θ)
- Time of flight: t = 2*vy/g
- Range: R = vx * t
</model>
<code>
import math
v = 20       # m/s
theta = 30   # degrees
g = 9.81     # m/s^2

vx = v * math.cos(math.radians(theta))
vy = v * math.sin(math.radians(theta))
t_flight = 2 * vy / g
range_m = vx * t_flight

print(f"vx = {vx:.3f} m/s")
print(f"vy = {vy:.3f} m/s")
print(f"Time of flight = {t_flight:.3f} s")
print(f"Range = {range_m:.3f} m")
</code>
<output>
vx = 17.321 m/s
vy = 10.000 m/s
Time of flight = 2.039 s
Range = 35.304 m
</output>
The projectile travels approximately **35.3 meters**. It's in the air for about 2 seconds.
```

### 2.4 Example: Using `<model>` for Design (no computation needed)

```
User: How would you design a rate limiter?

<think>
This is an architecture question, not a computation.
Token bucket or sliding window are the standard approaches.
</think>
<model>
Service RateLimiter {
  state: {buckets: map<ClientId, TokenBucket>},
  methods: {
    checkAndConsume(clientId, cost) -> bool {
      preconditions: [client known or created],
      postconditions: [tokens decremented if allowed]
    }
  }
}
Type TokenBucket {
  tokens: float, capacity: int,
  refillRate: float, lastRefill: timestamp
}
</model>
A token bucket rate limiter maintains a counter per client that refills at a fixed rate. Each request costs tokens; when the bucket is empty, requests are rejected. This allows bursts up to the bucket capacity while enforcing a long-term average rate.
```

## 3. Execution Engine

### 3.1 Architecture

The execution engine wraps model generation and intercepts after each `</code>` token:

```
prompt → [model generates tokens] → </code> detected
→ extract code block
→ execute Python
→ inject <output>...</output> into context
→ [model continues generating]
→ repeat if another <code> appears
→ </s> (end of sequence)
```

### 3.2 Python Executor

**Two modes:**

**Inline (`exec()`)** — default for most cases:
- Maintains a persistent namespace dict across code blocks within one response
- Variables defined in block 1 are available in block 2
- Hard timeout: 10 seconds per block
- Captures stdout/stderr
- No file system access (sandbox via restricted builtins if needed)

**VM (scratchpad QEMU)** — for complex tasks:
- Full Python environment with pip packages
- File system access within VM
- Use when code needs: file I/O, network, specific libraries, subprocess
- Located at `~/workspace/scratchpad/`
- Slower (~seconds startup), use deliberately

### 3.3 Shared State

Within a single response, Python state persists across `<code>` blocks:

```python
# Block 1
data = [1, 4, 9, 16, 25]
print(sum(data))
```
```
# output: 55
```
```python
# Block 2 — data is still in scope
avg = sum(data) / len(data)
print(avg)
```
```
# output: 11.0
```

State resets at the start of each new user turn.

### 3.4 Tool Request Protocol

Every code block executes with two built-ins in its namespace:

- **`use_tool(name, **kwargs)`** — call an external tool by name
- **`ToolNotAvailableError`** — raised when the tool is not registered

```python
# Model can call any named tool
try:
    result = use_tool("web_search", query="current EUR/USD rate")
    # use result...
except ToolNotAvailableError as e:
    print(f"Need tool: {e}")
    # fall back gracefully
```

**Runtime registers tools before execution:**

```python
executor = PythonExecutor()
executor.register_tool("web_search", my_search_fn)
executor.register_tool("sql_query",  my_db_fn)
```

**Default behaviour (no tools registered):** `ToolNotAvailableError` is raised with a clear message listing what is available. The model is trained to catch this, explain what it needs, and offer an alternative (fallback computation, sample data, or a description of the query it would run).

**Why not a new token?** Using a Python call keeps everything inside the existing `<code>/<output>` execution flow. The runtime decides which tools to provide; the model decides when to use them. No new tokens, no extra parsing.

**Training behaviour taught:**
1. Probe tools when uncertain (`try: use_tool(...) except ToolNotAvailableError: ...`)
2. Explain clearly what tool is needed and why
3. Provide a useful fallback — never silently fail or hallucinate results
4. Discover available tools at runtime when the capability set is unknown

### 3.4 Error Handling

If code execution fails:
```
<output>
ERROR: NameError: name 'undefined_var' is not defined
</output>
```
The model sees the error and can recover:
```
<think>
The variable wasn't defined yet. Let me fix that.
</think>
<code>
undefined_var = 42
print(undefined_var)
</code>
<output>
42
</output>
```

## 4. Training Data

### 4.1 Format

JSONL, one example per line:

```json
{
  "id": "arith_001",
  "category": "arithmetic",
  "difficulty": "basic",
  "query": "What is 15% of 240?",
  "response": "<think>\nPercentage calculation: multiply by 0.15\n</think>\n<code>\nresult = 240 * 0.15\nprint(f'15% of 240 = {result}')\n</code>\n<output>\n15% of 240 = 36.0\n</output>\n15% of 240 is **36**."
}
```

**Critical requirement**: All `<output>` blocks must be pre-executed and verified. No hallucinated outputs.

### 4.2 Categories

```
training/datasets/
├── arithmetic/          # Basic math operations, percentages, fractions
├── algebra/             # Equations, inequalities, polynomials
├── statistics/          # Mean, median, std dev, distributions
├── science/             # Physics, chemistry, biology calculations
│   ├── physics/
│   ├── chemistry/
│   └── biology/
├── programming/         # Algorithms, data structures, sorting, searching
├── finance/             # Compound interest, NPV, amortization
├── geometry/            # Area, volume, trigonometry
├── logic/               # Multi-step reasoning problems
├── design/              # Think+model only (no code) — system design
└── multi_step/          # Complex problems requiring multiple code cycles
```

### 4.3 Complexity Tiers

**Tier 1 — Single execution** (most examples):
- One `<think>` → optional `<model>` → one `<code>/<output>` → answer

**Tier 2 — Multi-step**:
- Two or three `<code>/<output>` cycles
- Each cycle uses results from previous
- Cover: iterative refinement, intermediate calculations feeding final answer

**Tier 3 — Error recovery**:
- First code block has a bug or wrong approach
- Model sees error output, reasons about it, corrects
- Demonstrates real-time learning from execution feedback

**Tier 4 — Design only** (no code):
- `<think>` + `<model>` (Blueprint notation) + prose answer
- Used for: architecture questions, conceptual explanations, planning
- Preserves the Blueprint design capability

### 4.5 Automated vs Manual

**Automated** (script-generated + verified):
- Arithmetic, algebra, geometry: generate problems programmatically, execute Python to get answers, build examples
- Statistics: generate datasets, compute answers

**Manual/curated**:
- Physics, chemistry: need physically meaningful setups
- Multi-step logic: need narrative coherence
- Design: need expert system architecture knowledge
- Error recovery: need plausible mistakes and corrections

## 5. Model and Training

### 5.1 Base Model

**Qwen3-1.7B** — justification:
- 1.7B is meaningfully better than 0.6B at generating correct Python
- Fits comfortably in 32GB VRAM with LoRA training
- Well-pretrained on code (Python coverage is good)
- Already validated in this project's infrastructure

### 5.2 Training Approach

**Method**: LoRA fine-tuning (same as previous phases, proven to work)

**Hyperparameters** (from TRAINING_COMPARISON.md learnings):
- Epochs: 10–15
- Batch size: 2, gradient accumulation: 8 (effective batch: 16)
- Learning rate: 2e-4 with cosine decay
- Warmup: 10% of steps
- Context length: 1024 tokens (longer than before — code blocks are verbose)
- LoRA rank: 16, alpha: 32

**Token handling**:
- Add `<think>`, `</think>`, `<model>`, `</model>`, `<code>`, `</code>`, `<output>`, `</output>` as special tokens
- Train model to predict all tokens including inside `<output>` blocks — this is how it learns what outputs look like
- At inference time, we intercept and inject real outputs instead

**Loss masking**:
- Standard next-token prediction loss on full sequence
- Consider masking `<output>` content loss (model shouldn't be penalized for not knowing output — it will be injected). This is a design choice to validate empirically.

### 5.3 Hardware

- AMD Radeon MI50 (32GB VRAM)
- ROCm 7.2 (see `~/workspace/rocm/7.2.0/`)
- Training script must use ROCm-compatible PyTorch

## 6. Inference Loop

### 6.1 Custom Generation

Standard `model.generate()` is insufficient — we need to intercept mid-sequence.

```python
# Pseudocode
def generate_with_execution(model, tokenizer, prompt, executor):
    context = tokenize(prompt)
    namespace = {}  # shared Python state for this response

    while True:
        tokens = model.generate(context, stop_at=["</code>", "</s>"])
        context += tokens

        if ends_with("</s>"):
            break

        if ends_with("</code>"):
            code = extract_last_code_block(context)
            stdout, stderr = executor.run(code, namespace)
            output_text = stdout or stderr or "(no output)"
            output_tokens = tokenize(f"\n<output>\n{output_text}\n</output>\n")
            context += output_tokens

    return detokenize(context)
```

### 6.2 Stopping Tokens

Model must be trained to stop generating after `</code>` and "wait" for output. In practice: the generation loop uses `</code>` as a stop token for each generation step.

## 7. Evaluation

### 7.1 Metrics

**Execution success rate**: % of generated code blocks that run without error
**Output accuracy**: For examples with known answers, % of outputs matching expected value
**Token presence**: % of responses using think/code/output when appropriate
**Reasoning coherence**: Manual review — does the model use the output correctly?

### 7.2 Test Suite

Maintain a held-out test set (~10% of each category) with known answers. Automated evaluation: run inference, execute code, compare output to expected.

## 10. Open Questions

1. **Loss masking on `<output>`**: Should we mask the loss on output tokens so the model learns "output comes from execution, not generation"? Or train on full sequence so the model understands what outputs look like? Test empirically in Phase 2.

2. **`<model>` tag scope**: Currently flexible (phrase to full Blueprint). Should it have a preferred format? Probably keep flexible — let training data show variety.

3. **VM vs inline execution**: For training data generation, use inline exec(). For inference, use inline exec() by default with fallback to scratchpad VM. Complex multi-turn tasks might need the VM.

4. **Error recovery training**: How many error-recovery examples vs clean examples? Too many errors → model generates buggy code. Too few → model doesn't handle errors. Target ~15% error-recovery in final dataset.
